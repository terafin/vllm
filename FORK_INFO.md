# Fork tracking — intarweb/vllm

This is a soft fork of [`vllm-project/vllm`](https://github.com/vllm-project/vllm). We track upstream HEAD and publish a HEAD-tracked image to GHCR with any open intarweb→upstream PRs cherry-picked on top.

> All forks we manage with the sync-upstream + from-source-build pattern live under the [`intarweb`](https://github.com/intarweb) GitHub org. See the `ghcr-fork-mirror` skill for the canonical recipe.

## Branch model

| Branch | Purpose | Source of truth |
|---|---|---|
| `main` | Upstream-clean tracking + ops overlay (canonical workflows + FORK_INFO.md) | upstream + us |
| `intarweb-dev` | Deploy track — `main` + cherry-pick of every open intarweb→upstream PR. Drives `:latest`. | us |
| `feat/*`, `fix/*` | Individual patch branches we PR upstream | us |

## How to consume

```yaml
# docker-compose.yml
services:
  vllm-openai:
    image: ghcr.io/intarweb/vllm-openai:latest
```

Pin to a specific build for reproducibility:

```yaml
    image: ghcr.io/intarweb/vllm-openai:sha-<short>
```

## Maintenance recipes

**Manually re-sync onto upstream:**
```bash
gh workflow run "Sync from upstream + auto-regen intarweb-dev" --repo intarweb/vllm
```

**Force a fresh build of `:latest` without an upstream change:**
```bash
gh workflow run "Build from source → GHCR" --repo intarweb/vllm --ref intarweb-dev
```

**Add a new patch to intarweb-dev:**
```bash
git checkout -b feat/my-thing upstream/main
# ...do work...
git push origin feat/my-thing
gh pr create --repo vllm-project/vllm --base main --head intarweb:feat/my-thing
# Next sync-upstream tick auto-cherry-picks the new PR onto intarweb-dev.
```
